## Carryover for P4 (wiring layer)

P4 owns mitigations for surfaces P3 deliberately excluded:

1. **Adapter-boundary input validation (T-01)** — Run zod `safeParse` against CEMADEN/INMET responses before constructing `Alert[]` for the engine. Reject (don't coerce) malformed entries; log + skip. Engine assumes typed input — feeding it untyped data may produce `undefined` lookups in vocab maps yielding fallback strings (`source_key` raw) but will not crash; still, validation is required for data quality.
2. **`/api/ingest` token auth + replay window** — Out of P3 scope. Existing P2 token guard applies; P4 must re-verify after wiring.
3. **`/api/archive` cron auth** — Same. Verify Bearer header reaches handler post-redirect (see MEMORY note: apex→www auth header drop — use canonical `www.ensobrasil.com.br`).
4. **Snapshot consumer escaping (T-05)** — UI renderers (P5) must HTML-escape `explanation` string when injecting into DOM. Current copy is enum-bounded but defense-in-depth still warranted.
5. **Time consistency (T-08)** — Pass a single `new Date()` into both `calculateRiskLevel` and `applyStaleness` for one snapshot computation; do not call them with independent `new Date()` reads.
6. **Rate limiting at `/api/ingest` (T-02)** — Cap `Alert[]` size or per-IP/per-token request rate. Engine is robust to ~1k alerts/state but adversarial input deserves a guard.
7. **Sources-health row trust (T-07)** — `applyStaleness` trusts `last_successful_fetch` strings as ISO-8601. P4 must ensure this column is set only by trusted adapter code, never user input.
